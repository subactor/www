Subactor vs Agent Contracts

Agent Contracts najprecyzyjniej formalizuje granice zasobów autonomicznego agenta. Subactor używa szerszego pojęcia kontraktu: obejmuje ono odpowiedzialność, operacje, procesy i dowody dla ludzi oraz maszyn.

Werdykt

To projekty komplementarne. Agent Contracts jest wyspecjalizowaną biblioteką budżetów i lifecycle agentów; Subactor jest control plane'em organizacji i wykonania. Największą wartością integracji byłoby dodanie formalnego budżetu do każdego uruchomienia URI Process.

Co formalizuje Agent Contracts

Kontrakt zawiera specyfikację wejścia i wyjścia, umiejętności, ograniczenia zasobów, granice czasu, kryteria sukcesu i warunki zakończenia. Lifecycle przechodzi od DRAFTED do ACTIVE, a potem do jednego ze stanów terminalnych: FULFILLED, VIOLATED, EXPIRED albo TERMINATED. Delegowanie ma zachowywać budżet kontraktu nadrzędnego.

Najważniejsza różnica

Agent Contracts odpowiada na pytanie „w jakich granicach obliczeniowych agent może próbować?”. Subactor odpowiada również „czy ten aktor ma uprawnienie, do jakiej odpowiedzialności należy zadanie i czy efekt rzeczywiście pojawił się w organizacji?”. Sam budżet nie definiuje procesu ludzi, kanału eskalacji ani źródła prawdy rekordów.

Kiedy wybrać lub połączyć

  • Agent Contracts: gdy najważniejsze są koszty LLM, zapobieganie pętlom, terminy i formalne warunki zakończenia agenta.
  • Subactor: gdy kontrakt ma obejmować organizację, role, operacje, stan biznesowy i dowód wykonania.
  • Razem: przełożyć budżet Agent Contract na ograniczenia procesu Subactor i zapisywać końcowy stan kontraktu w receipcie.

Status „production-ready” pochodzi z deklaracji autorów repozytorium. Traktujemy projekt jako wczesny OSS i nie utożsamiamy deklarowanych testów z niezależną walidacją produkcyjną.
